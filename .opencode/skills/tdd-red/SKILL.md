# tdd-red — Write the failing test

The RED phase produces **one focused failing test** for the next unmet behavior. Correctness
expectations come from vectors generated by the reference JS `color-convert` library — never from
hand-computed numbers (see `AGENTS.md` Rule 8).

## When to use

Invoked by `red-dev` (or the `orchestrator`) at the start of each TDD cycle, before any
implementation exists for the behavior under test.

## Procedure

1. **Identify the single next behavior.** From the issue's behavior list, pick the simplest one
   not yet covered by a passing test. One behavior per cycle — never batch.
2. **Locate the reference vector.** Use a committed vector file under `tests/vectors/` (generated
   by `benchmarks/js/gen-vectors.mjs`). If the needed vector does not exist yet, that vector
   generation is its own issue — do not invent numbers.
3. **Write one test.** Assert the conversion output against the reference within a documented
   tolerance. Name it descriptively: `rgb_to_hsl_matches_reference_pure_red`.
4. **Run it:** `cargo test <test_name>`.
5. **Confirm the RIGHT failure.** It must fail on the assertion or a missing function — not on an
   unrelated compile error elsewhere. If a stub is required just to compile, keep it a
   `unimplemented!()` placeholder and leave real logic to GREEN.
6. **Commit:** `test(<scope>): add failing test for <behavior> 🔴`.
7. **Report** to the orchestrator: behavior, vector source, expected value, exact failure message.

## Test shape (guidance, authored during the coding session)

```rust
// tests/rgb_hsl.rs
#[test]
fn rgb_to_hsl_matches_reference_pure_red() {
    // vector: [255,0,0] -> [0, 100, 50]  (source: tests/vectors/rgb_hsl.json)
    let got = color_convert_rs::rgb::to_hsl([255, 0, 0]);
    let want = [0.0, 100.0, 50.0];
    for (g, w) in got.iter().zip(want) {
        assert!((g - w).abs() <= 1e-3, "got {got:?}, want {want:?}");
    }
}
```

## Tolerance policy

- State the tolerance inline and why (float rounding, integer output, etc.).
- Do NOT loosen a tolerance to make a future GREEN easier. Tolerance mirrors the JS library's
  observable precision, nothing more.

## DO / DON'T

| DO | DON'T |
|----|-------|
| Write exactly one test | Batch multiple behaviors |
| Source expectations from JS vectors | Hand-fabricate expected numbers |
| Confirm the failure reason | Accept a spurious compile error as "red" |
| Keep any needed stub trivial | Implement the conversion here |
